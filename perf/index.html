上面的heap实际上只能用来分析堆内存，熟悉栈和堆区别的老哥们可以知道，Go编译器在编译期间对每个变量直接做了“逃逸”分析，对于有可能在函数返回后仍然可用的对象，它就会被分析为逃逸的变量，所以它的数据被存储在堆上，所以Golang是完全不用理解栈和堆这些概念的。

那么问题来了，为什么上面我分配了一个明显不逃逸的变量，它还是能被heap分析？结论是和它的容量有关，我这里测试大概大于2m时在堆上分配，如果较小上面的bytes就在栈上分配，heap就看不到了。

这里简单解释一下为什么栈上不适合放太大的对象。在C/C++这样的语言中，的栈大小是固定的(比如4m)，如果超过了就stack overflow。但是Go其实是动态栈，如果超过了它会自动增长，即便如此它也不适合存大对象，原因有几个：一个是栈增大可能触发栈扩容，此时是必须新分配一块栈，进行拷贝的，代价太大；第二个是GC问题，大的对象可能意味着里面有指针，GC会对栈内的对象使用三色标记法做可达性分析，这里很有可能会增加扫描时间。

综上所述我们对栈/堆，逃逸分析，甚至是GC有了更深刻的理解：

  • slice变量本身只是一个很小的header，包含data pointer、len、cap。这个 header 作为局部变量时通常可以放在栈上，甚至被优化到寄存器里。make([]T, n) 创建的backing array放在栈上还是堆上，不只取决于逃逸分析，还取决于n是否编译期可知、对象大小是否超过编译器允许的栈分配阈值等。这个决策主要发生在编译期。
  • 对于 map、channel这类引用类型，栈上存的局部变量本身通常只是一个很小的引用值；真正承载状态的数据结构，例如map的 hmap/buckets、channel的hchan/buffer，通常由runtime在堆上分配和管理。
  • GC Root主要包含栈，全局变量，它会使用三色标记一层层追溯，做可达性分析，如果栈帧/全局变量里有很多指针如var arr [1_000_000]*Object，或者是对象和对象之间使用指针有形成了很深的引用链。这个时候垃圾回收器就会mark很久占用更多cpu。这个被成为根集合过大。

最后可以讨论下如何解决根集合过大的问题，可以看看下面的业务场景：

在上面的业务场景中，我们用一个map存储了大量数据，UserMap是一个根对象，GC扫描每个kv一个个往下做可达性分析，这里主要扫描的是UserMap内的string key和*User value。string实际上是一个string header，算一个指针，指向底层真实的数据；*User本身就是一个指针，指向一个堆User对象，并且它下面还有两个string(记得它们是string header，看起来就和指针一样)。所以如果有100个kv，一次gc基本上要扫描约400个指针槽。

优化方法：

  1. 使用map[string]User，这样实际上可以理解为User下面的两个string header指针被直接放在map bucket里面，少了一层指针，现在只用扫描300次。
  2. User内部用UserId uint64这样的数值类型，少用指针或者其它引用类型。
  3. 把User编码成json，UserMap变成map[string]string，这样只用扫描200次了。
  4. 如果有些内容长度固定或者一定小于某个长度，使用数组把字段弄成一个值Password [32]byte，这样就规避指针扫描了。
  5. 从根源上解决做冷热分离，把不太活跃的用户淘汰到文件上，内存里存少一点，但是这其实是在逃避我们上面所讨论的问题。
  6. 数据挂外面去，比如使用boltdb，leveldb。或者直接使用freecache、bigcache库，它们就是专门用来解决cache场景下large gc root问题的，如果使用这种cache它会比map[string]string更狠，直接干到几乎0GC。
